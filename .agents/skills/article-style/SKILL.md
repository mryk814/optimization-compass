# 記事スタイルガイド

このskillは `content/**/*.md` の**文章品質と表記統一**のルールを定める。
構造ルール（frontmatter、許可されるMarkdown構文、検証手順）は
[`docs/content-authoring.md`](../../../docs/content-authoring.md) が正で、このskillはそれに文体レイヤーを追加する。

## 0. 推敲時の不変条件（最重要）

文章を整えるとき、次は**絶対に変更しない**。

- 技術的事実、数値、主張の強さ（保証・限界の記述を強めたり弱めたりしない）
- frontmatterの全フィールド。ただし `summary`（本文第1段落と完全一致を維持するため連動して更新）と `last_reviewed`（推敲日に更新）は除く
- コードブロックの中身、数式（`$...$` / `$$`）、リンク先URL・ルート（`#/...`）、図の参照
- 見出しの階層構造（`##` 始まり、レベル飛ばし禁止）
- callout種別（`::: note` / `::: tip` / `::: warning`）の意味的な使い分け
- 出典が必要になる新しい事実は**追加しない**（推敲は言い換えであって加筆ではない）

「ranking・普遍的推奨を作らない」「保証を無条件化しない」というリポジトリのcore invariantは文レベルでも守る。

### 言語contract

公開教材は [`docs/adr/0013-japanese-first-language-strategy.md`](../../../docs/adr/0013-japanese-first-language-strategy.md)
に従い、**日本語を説明の主言語とし、英語の正式用語も見える・探せる**形で執筆する。

- `status: published`の記事は、`title_ja`、`summary`、本文だけで日本語の説明を完結させる。
- `title_en`、英語alias、略語は正式用語・検索metadataであり、英語版記事として扱わない。
- 手法名、製品/API名、code、数式、source titleは正しい原表記を保つ。
- stable ID、URL identifier、schema key、API fieldは翻訳しない。
- 未レビューの機械翻訳を、執筆済みの翻訳や欠落localeのfallbackとして追加しない。

## 1. 文体

- **です・ます調**で統一する。体言止めの本文段落は使わない（リスト・表内は除く）。
- **一文一義**。読点3つ以上または90字を超える文は分割を検討する。
- 冗長表現は圧縮する:
  - 「〜することができます」→「〜できます」
  - 「〜という〜」の連続、「〜となっています」→「〜です」
  - 「基本的に」「非常に」「しっかり」などの程度語は、判断基準を示せないなら削る
- 主語の省略は可。ただし係り受けが2通りに読める文は書き直す。
- 断定は根拠の範囲で行う。保証がない事柄は「〜する場合があります」「〜とは限りません」の形で、曖昧化ではなく条件を添えて書く。
- 読者は「手法を学ぶ人」であり、ソルバー実装者ではない（`docs/method-content-density.md` のbeginner-first原則）。数式は、式が何を表すかを先に日本語で述べてから使う。

## 2. 用語・表記

### 2.1 英語で書く語（ASCII小文字、カタカナ化しない）

実装・診断・実務の用語は英語表記が正準:

step / step length / scale / scaling / memory / model / solver / budget /
oracle / line search / trust region / gradient norm / ranking / family /
warm start / restart / bounds / tolerance / black-box / surrogate /
best-so-far / feasible / infeasible

- ✕ ステップ、スケール、メモリ、モデル、ソルバー、ランキング
- 「〜の回数」は「function evaluation数」「iteration数」のように **英語名詞+数** で書く（診断値リストで特に統一）。

### 2.2 日本語で書く語

数学的な中核概念は日本語が正準:

勾配、曲率、目的関数、制約、凸性、局所解、大域最適、収束、反復（地の文）、
初期点、停止条件、探索方向、正定値

- ✕ gradient（地の文の単独名詞として）。ただし「gradient norm」「gradient check」「gradient oracle」など複合語は英語のまま。

### 2.3 固有名

- 手法名は正式英語名: BFGS、Nelder–Mead（en dash）、L-BFGS-B、Gauss–Newton、CMA-ES など。既存の `title_en` に合わせる。
- 「Newton法」「勾配降下法」のように定着した和名はそのまま使う。記事内で和名と英名を混在させない（初出でどちらかに決める）。

## 3. 見出しの正準形

### 3.1 共通の置換表

同じ役割の見出しは次に正規化する:

| 揺れている表記 | 正準 |
|---|---|
| Python例 | Python |
| 見るべき診断値 | 診断値 |
| 失敗の兆候 / 失敗したときに確認する | 失敗・切替の兆候 |
| 向いている問題 | 向いている条件 |
| 向いている／避ける条件 | 向く条件・避ける条件 |
| 次に見る | 次に読む |

「向く条件」と「避ける条件」を**別セクション**に分ける場合は「向いている条件」「避ける／切り替える条件」の対で書く。**一体**で書く場合は「向く条件・避ける条件」とする。

### 3.2 method記事（標準型）

構成順序:

1. 導入段落（frontmatter `summary` と完全一致。カード・SEO説明文として単独で成立させる）
2. 仕組みの説明 1〜3セクション（見出しは内容に即した自由題。「単体は何を表すか」のような具体的な問いが良い）
3. 「向いている条件」（必要なら「避ける／切り替える条件」）
4. 「Python」（コード直後に、結果の読み方と公式リファレンスへの誘導）
5. 「診断値」
6. 「失敗・切替の兆候」
7. 締め: 関連手法へのリンク段落、または「次に読む」

### 3.3 method/family記事（選び分け型）

`30秒でつかむ` テンプレートを使う記事は `docs/method-content-density.md` の構成に従う:

30秒でつかむ → まず確認すること → 条件付きの選び分け →
うまくいったサインと切替サイン → 小さな比較の型 → コラム: 〜 → 次に読む

どちらの型を使うかは既存記事のまま維持する。推敲で型を移植しない。

### 3.4 concept記事

「直感」で始め、定義 → 最適化での意味 → 注意（callout）の順を基本とする。

## 4. リスト・表・callout

- 1つのリスト内では文体を統一する（全項目を名詞止め、または全項目を文）。
- 表のセルは名詞句を基本とし、「〜ため」で理由を書く列は全行で語尾を揃える。
- calloutは1記事あたり合計2個まで。`warning` は「読者が実害を被る誤用」に限定する。
- 「これは一般性能rankingではありません」のような公平性の但し書きは削らない。

## 5. 推敲チェックリスト

推敲を終える前に確認する:

- [ ] 第1段落と frontmatter `summary` が完全一致している
- [ ] `last_reviewed` を更新した
- [ ] 2.1 / 2.2 の表記ゆれが残っていない（例: ステップ、モデル、勾配ノルム→gradient norm）
- [ ] 3.1 の見出し置換表を適用した
- [ ] 事実・数値・リンク・コード・数式を変えていない
- [ ] 検証: `uv run python scripts/verify_content.py`（Tier A）
