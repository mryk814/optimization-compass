# Migrating dataset 0.2.0 to 0.3.0

## Consumers

No destructive table or column migration is required. Consumers that only read the existing
recommendation tables can replace the SQLite file and continue using the same API/CLI contracts.
Consumers that validate a fixed `dataset_version` must accept `0.3.0` explicitly and re-export
their cached site data.

## Static Atlas

Replace the complete `site/public/data` directory from the v0.3.0 staged tree. Do not mix v0.2.0
ViewSpec, recommendation, content, gallery, comparison, or Trace assets with v0.3.0 assets.
`release.json`, `manifest.json`, and every generated asset must agree on `dataset_version`.

Persisted/shareable AtlasState created against v0.2.0 is a versioned input. Decode it through the
existing compatibility boundary and show migration warnings where applicable; never relabel it as
v0.3.0 without validation against the new ViewSpec.

## Maintainers

1. Edit only `src/optimization_compass/resources/release-authority.json` for the next version/date.
2. Run `--stage --output <directory>` and retain the reported tree SHA-256.
3. Review the versioned manifest and release notes.
4. Run `--publish --staged-directory <directory>`.
5. Run the full Python/site/parity validation and confirm generated-data drift is zero.
6. Tag and publish release assets only from the validated main commit.
